# 最多能参加比赛数

# 最多能参加比赛数 深入浅出解题教案

## 1. 题目深度解析
### - **核心考点**
1.  **贪心算法 (Greedy Algorithm)**：本题是经典的“活动选择问题” (Activity Selection Problem)，是贪心算法的典型应用。它要求我们在面对多个选择时，每一步都做出局部最优的选择，最终期望得到全局最优解。
2.  **区间问题**：题目中的比赛可以看作是时间轴上的一个个区间 $[a_i, b_i]$，我们需要选择互不重叠的区间，使其数量最多。
3.  **排序**：为了有效地应用贪心策略，通常需要对数据进行特定的排序。

### - **题目难点**
1.  **贪心策略的选择与证明**：面对多个可能的贪心策略（例如，按开始时间排序、按持续时间排序、按结束时间排序），如何判断哪种策略是正确的，并能直观地理解其正确性。这对于初学者来说是难点。
2.  **边界条件处理**：比赛结束的瞬间是否可以立即开始下一场比赛？题目中“不能同时参加两场或以上的比赛”暗示只要前一场比赛结束，下一场比赛就可以开始。例如，$[0,2]$ 和 $[2,4]$ 是可以连续参加的。

### - **易错点警示**
1.  **排序错误**：如果不对比赛进行正确的排序，或者排序的依据选择错误，将无法得到正确答案。例如，仅仅按开始时间排序往往是错误的。
2.  **判断条件错误**：在选择下一场比赛时，判断条件 $a_i \ge \text{last\_end\_time}$ 容易写成 $a_i > \text{last\_end\_time}$。如果写成严格大于，那么像 $[0,2]$ 和 $[2,4]$ 这种紧密相连的比赛就无法连续参加了，导致结果偏小。
3.  **数据范围**：$N$ 最大可达 $10^6$，这意味着 $O(N^2)$ 甚至 $O(N \log N \log N)$ 的算法都可能超时。我们需要一个高效的 $O(N \log N)$ 或 $O(N)$ 的解法。

## 2. 教学目标
通过学习本题，学生应该能够：
1.  理解贪心算法的基本思想，并认识到它在解决某些优化问题中的高效性。
2.  掌握“活动选择问题”的标准贪心解法，并能解释其核心原理。
3.  学会如何分析和选择合适的排序策略来辅助贪心算法。
4.  提升对区间类问题的处理能力，包括如何判断区间冲突、如何选择不冲突的区间。
5.  培养从暴力解法逐步优化到高效解法的思维过程。

## 3. 循序渐进的解题思路

### 阶段一：直观想法（暴力/朴素解法）

首先，我们来思考最直接、最容易想到的方法。题目要求我们从 $N$ 场比赛中选出尽可能多的比赛，且这些比赛不能相互冲突。

**思路**：
既然要选出所有可能的组合，我们可以尝试枚举所有比赛的子集，然后对每个子集判断其是否合法（即所有比赛都不冲突），并记录最大合法的子集大小。

**具体实现**：
我们可以使用**回溯法**或**深度优先搜索 (DFS)** 来实现。
1.  定义一个递归函数 `dfs(index, current_selected_matches, last_end_time)`。
2.  在 `dfs` 函数中，对于当前比赛 `index`，有两种选择：
    *   **选择参加**：如果当前比赛的开始时间 $a_{index}$ 大于等于 `last_end_time`，则可以选择参加。将 `current_selected_matches` 加 $1$，更新 `last_end_time` 为 $b_{index}$，然后递归调用 `dfs(index + 1, ...)`。
    *   **选择不参加**：直接递归调用 `dfs(index + 1, ...)`，不改变 `current_selected_matches` 和 `last_end_time`。
3.  在递归的基线条件 (所有比赛都考虑完毕) 中，更新全局的最大比赛数。

**时间复杂度分析**：
对于每场比赛，我们都有“参加”或“不参加”两种选择。总共有 $N$ 场比赛，所以理论上会有 $2^N$ 种选择组合。即使在递归过程中进行了剪枝，最坏情况下仍然接近 $O(2^N)$。

**为什么它可能无法通过（TLE/MLE）**：
题目中 $N$ 的数据范围高达 $10^6$。$2^{10^6}$ 是一个天文数字，即使 $N=20$，$2^{20}$ 也已经超过一百万。因此，这种指数级的暴力解法会毫无疑问地超时 (Time Limit Exceeded, TLE)。

### 阶段二：优化突破口

暴力解法行不通，我们需要寻找更高效的方法。这类“选择不冲突区间”的问题，往往是贪心算法的用武之地。但如何贪心呢？我们来尝试几种直观的贪心策略。

**核心思想**：贪心算法的每一步都做出当前看起来最优的选择，希望最终能导致全局最优。

**尝试一：按开始时间 $a_i$ 排序**
我们把所有比赛按照开始时间 $a_i$ 从小到大排序。然后从第一场比赛开始，如果能参加就参加，然后继续选择下一场能参加的比赛。
*   **例子**：比赛有 `(0, 10), (1, 2)`。
*   按开始时间排序：`(0, 10), (1, 2)`。
*   选择 `(0, 10)`。现在结束时间是 $10$。
*   下一场 `(1, 2)`，开始时间 $1 < 10$，不能参加。
*   总共参加 $1$ 场。
*   **问题**：如果选择 `(1, 2)`，虽然开始时间晚一点，但它结束得早，可能会为后续更多的比赛腾出时间。例如，如果还有一场 `(3, 4)`，那么选择 `(1, 2)` 后，还可以选择 `(3, 4)`，总共 $2$ 场。而选择 `(0, 10)` 就只能参加 $1$ 场。
*   **结论**：仅仅按开始时间排序不是最优策略。

**尝试二：按比赛持续时间 $(b_i - a_i)$ 排序**
我们优先选择持续时间最短的比赛，因为它们占用时间最少，可能为其他比赛留下更多空间。
*   **例子**：比赛有 `(0, 100), (0, 10), (11, 20)`。
*   持续时间：`(0, 10)` 持续 $10$；`(11, 20)` 持续 $9$；`(0, 100)` 持续 $100$。
*   按持续时间排序（从小到大）：`(11, 20), (0, 10), (0, 100)`。
*   选择 `(11, 20)`。现在结束时间是 $20$。
*   下一场 `(0, 10)`，开始时间 $0 < 20$，不能参加。
*   下一场 `(0, 100)`，开始时间 $0 < 20$，不能参加。
*   总共参加 $1$ 场。
*   **问题**：如果先选择 `(0, 10)`，然后结束时间是 $10$。下一场 `(11, 20)`，开始时间 $11 \ge 10$，可以参加。总共 $2$ 场。
*   **结论**：按持续时间排序也不是最优策略。

**尝试三：按结束时间 $b_i$ 排序**
我们把所有比赛按照结束时间 $b_i$ 从小到大排序。然后从第一场比赛开始，如果能参加就参加，然后继续选择下一场能参加的比赛。
*   **例子**：比赛有 `(0, 2), (2, 4), (1, 3)`。
*   按结束时间排序：
    1.  `(0, 2)` (结束时间 $2$)
    2.  `(1, 3)` (结束时间 $3$)
    3.  `(2, 4)` (结束时间 $4$)
*   **步骤**：
    1.  初始化 `count = 0` (已参加比赛数)，`last_end_time = -1` (上一次参加比赛的结束时间，设为足够小以确保第一场能被选)。
    2.  遍历排序后的比赛列表：
        *   考虑比赛 `(0, 2)`：开始时间 $0 \ge -1$。可以参加。`count = 1`，`last_end_time = 2`。
        *   考虑比赛 `(1, 3)`：开始时间 $1 < 2$。不能参加（与 `(0, 2)` 冲突）。
        *   考虑比赛 `(2, 4)`：开始时间 $2 \ge 2$。可以参加。`count = 2`，`last_end_time = 4`。
    3.  所有比赛遍历完毕，最终 `count = 2`。这与样例输出一致。

**关键逻辑推导：为什么按结束时间排序是正确的贪心策略？**
这个策略的正确性基于一个重要的贪心选择性质：**在所有可选的比赛中，优先选择最早结束的比赛，总是最优解的一部分。**

让我们来直观地理解一下：
假设我们已经选择了 $k$ 场比赛，并且它们都互不冲突。现在我们要选择第 $k+1$ 场比赛。
如果我们选择了一场结束时间比较晚的比赛 $X$，那么它会占据更长的时间轴，从而可能导致后续很多比赛都无法参加。
反之，如果我们选择了一场结束时间最早的比赛 $Y$（在当前可选择的比赛中），那么它会尽快地结束，从而**最大化地为后续比赛留出更多的时间空间**。
换句话说，最早结束的比赛，对后续选择的限制最小。这种“不挡路”的局部最优选择，最终能够帮助我们获得全局最优解。

**严谨证明 (交换论证法)**：
假设存在一个最优解 $S = \{A_1, A_2, \dots, A_k\}$，其中 $A_1, A_2, \dots, A_k$ 按照结束时间排序。
而我们的贪心策略选择的第一个比赛是 $G_1$，它是所有比赛中最早结束的。
1.  如果 $A_1 = G_1$，那么贪心策略和最优解的第一步是相同的，我们可以在剩下的比赛中继续寻找最优解。
2.  如果 $A_1 \neq G_1$，那么根据 $G_1$ 的定义，它的结束时间 $b_{G_1}$ 一定小于等于 $A_1$ 的结束时间 $b_{A_1}$。
    *   我们可以构造一个新的解 $S' = \{G_1, A_2, \dots, A_k\}$。
    *   因为 $G_1$ 结束时间更早（或相同），它与 $A_2$ 的开始时间 $a_{A_2}$ 冲突的可能性不会比 $A_1$ 更大。事实上，$b_{G_1} \le b_{A_1} \le a_{A_2}$ (因为 $A_1$ 和 $A_2$ 不冲突)，所以 $G_1$ 肯定也不会与 $A_2$ 冲突。
    *   因此，$S'$ 也是一个合法的解，并且它包含的比赛数量与 $S$ 相同。
    *   这意味着，存在一个最优解，它的第一个选择就是所有比赛中最早结束的比赛。
    *   有了这个“贪心选择性质”，结合“最优子结构性质”（即原问题的最优解包含子问题的最优解），我们就可以通过迭代地选择最早结束的比赛来构建全局最优解。

### 阶段三：最终算法流程

1.  **数据结构**：用一个结构体或 `std::pair<int, int>` 来存储每场比赛的开始时间和结束时间。为了方便排序，通常将结束时间放在前面，或者使用自定义比较函数。这里我们用 `std::pair<int, int>` 存储 `(b_i, a_i)`，这样 `std::sort` 默认会先按结束时间排序。
2.  **输入**：读取 $N$ 场比赛的开始时间和结束时间，并存储到 `std::vector` 中。
3.  **排序**：使用 `std::sort` 对存储比赛的 `std::vector` 进行排序。排序的依据是比赛的**结束时间** $b_i$ 从小到大。如果结束时间相同，可以按开始时间 $a_i$ 从小到大排序（这有助于保持稳定性，但对结果不影响）。
4.  **贪心选择**：
    *   初始化 `count = 0` (已参加的比赛数量)。
    *   初始化 `last_end_time = -1` (记录当前已参加的最后一场比赛的结束时间。设为 $-1$ 是为了确保第一场比赛总能被选中，因为所有 $a_i \ge 0$)。
    *   遍历排序后的比赛列表：
        *   对于当前比赛 `(a_i, b_i)`：
            *   如果当前比赛的开始时间 $a_i$ 大于等于 `last_end_time`，说明这场比赛可以参加，因为它不与之前已选择的任何比赛冲突。
            *   如果可以参加，则 `count++`，并将 `last_end_time` 更新为当前比赛的结束时间 $b_i$。
5.  **输出**：最终的 `count` 值就是最多能参加的比赛场次。

## 4. 核心代码讲解

```cpp
<!-- AC_CODE -->
#include <iostream> // 引入输入输出流库
#include <vector>   // 引入动态数组库
#include <algorithm> // 引入排序算法库

// 定义一个结构体来表示一场比赛
// 结构体比pair更具可读性，尤其当成员变量有特定含义时
struct Match
{
    int start; // 比赛开始时间
    int end;   // 比赛结束时间
};

// 自定义比较函数，用于std::sort
// 贪心策略的核心：按照比赛的结束时间进行升序排序
// 如果结束时间相同，可以按照开始时间升序排序，以保持稳定性，但对本题结果无影响
bool compareMatches(const Match& m1, const Match& m2)
{
    if (m1.end != m2.end)
    {
        return m1.end < m2.end; // 优先按结束时间升序
    }
    return m1.start < m2.start; // 如果结束时间相同，则按开始时间升序
}

int main()
{
    // 优化输入输出，提高程序运行效率
    // std::ios_base::sync_with_stdio(false) 解除C++标准库与C标准库的同步
    // std::cin.tie(NULL) 解除cin与cout的绑定，使得cin不会在每次输入前刷新cout
    std::ios_base::sync_with_stdio(false);
    std::cin.tie(NULL);

    int n; // 比赛场次
    std::cin >> n;

    std::vector<Match> matches(n); // 创建一个存储n场比赛的动态数组

    // 读取每场比赛的开始时间和结束时间
    for (int i = 0; i < n; ++i)
    {
        std::cin >> matches[i].start >> matches[i].end;
    }

    // 使用自定义比较函数对比赛进行排序
    // 核心贪心步骤：按结束时间从小到大排序
    std::sort(matches.begin(), matches.end(), compareMatches);

    int count = 0; // 记录最多能参加的比赛场次
    // last_end_time 记录当前已参加的最后一场比赛的结束时间
    // 初始化为-1，确保第一场比赛（即使开始时间为0）也能被选中
    int last_end_time = -1; 

    // 遍历排序后的比赛列表，应用贪心策略
    for (int i = 0; i < n; ++i)
    {
        // 如果当前比赛的开始时间 大于等于 上一场已参加比赛的结束时间
        // 说明当前比赛与已参加的比赛不冲突，可以参加
        // 注意：这里是 >=，表示如果上一场比赛在某个时刻结束，下一场比赛可以在同一时刻开始
        if (matches[i].start >= last_end_time)
        {
            count++; // 参加比赛，计数器加一
            last_end_time = matches[i].end; // 更新最后一场比赛的结束时间
        }
    }

    // 输出结果
    std::cout << count << std::endl;

    return 0; // 程序正常结束
}

```

## 5. 复杂度分析

### - **时间复杂度**
1.  **输入数据**：读取 $N$ 场比赛的开始和结束时间，需要 $O(N)$ 的时间。
2.  **排序**：对包含 $N$ 场比赛的 `std::vector` 进行排序。`std::sort` 的平均和最坏时间复杂度都是 $O(N \log N)$。
3.  **贪心选择**：遍历排序后的比赛列表一次，每次循环内部的操作（比较、赋值）都是常数时间 $O(1)$。因此，这一步需要 $O(N)$ 的时间。

**总时间复杂度**：$O(N) + O(N \log N) + O(N) = O(N \log N)$。
对于 $N=10^6$， $N \log N \approx 10^6 \times \log_2 10^6 \approx 10^6 \times 20 = 2 \times 10^7$，这个数量级的操作可以在 1 秒内完成，因此可以通过所有测试用例。

### - **空间复杂度**
1.  **存储比赛数据**：需要一个 `std::vector` 来存储 $N$ 场比赛的开始和结束时间。每场比赛存储两个整数，所以空间复杂度为 $O(N)$。
2.  **排序辅助空间**：`std::sort` 在内部通常会使用 $O(\log N)$ 或 $O(N)$ 的额外空间（例如，内省排序在最坏情况下可能需要 $O(\log N)$ 的栈空间，或者某些实现会使用 $O(N)$ 的临时空间进行归并排序）。

**总空间复杂度**：$O(N)$。
对于 $N=10^6$，存储 $10^6$ 个结构体（或 $2 \times 10^6$ 个整数）大约需要 $8 \times 10^6$ 字节 = $8$ MB 的内存，这在通常的内存限制（如 $256$ MB 或 $512$ MB）内是完全足够的。

## 6. 课后思考

1.  **变式一：如果每场比赛都有一个收益值，如何选择比赛使得总收益最大？**
    *   这不再是简单的贪心问题。因为选择一场高收益但结束时间晚的比赛，可能会导致损失更多场低收益但加起来总收益更高的比赛。这通常需要使用**动态规划 (Dynamic Programming)** 来解决。你可以尝试定义状态 $dp[i]$ 为考虑前 $i$ 场比赛所能获得的最大收益，并思考状态转移方程。

2.  **变式二：如果 kunkka 可以同时参加 $K$ 场比赛，最多能参加多少场？**
    *   这意味着 kunkka 有 $K$ 个“分身”或“资源”。当一场比赛结束时，其中一个分身就被释放了。这可以用**优先队列 (Priority Queue)** 来解决。将所有比赛按开始时间排序。遍历比赛时，将当前比赛的结束时间放入一个大小为 $K$ 的最小堆中。如果堆的大小超过 $K$，说明需要释放最早结束的比赛，如果当前比赛的开始时间大于堆顶元素（最早结束的比赛），则可以替换。这本质上是寻找最大不重叠区间集合的变种，但允许 $K$ 个重叠。

3.  **变式三：如果题目要求选择最少的比赛来覆盖所有给定的时间区间，又该如何处理？**
    *   这被称为“区间覆盖问题”。它与活动选择问题有相似之处，但贪心策略不同。通常会先按开始时间排序，然后选择一个能覆盖最远点的区间，并不断迭代。
